# macOS 支持分阶段实施文档

这份文档是给内部推进用的，不追求对外正式表达，目标是把思路、范围、优先级和关键技术路线讲清楚，方便下周直接开工。

## 1. 先说结论

我们不应该把“支持 macOS”理解成一次性把 Windows 全量能力搬过去。

更合理的节奏是：

1. 先做一个最小可运行版本，目标是浏览器自动化可用，80% 可跨平台组件可以正常运行。
2. 再把它补到可发布状态，解决权限、签名、打包、安装体验等问题。
3. 最后单独推进桌面自动化。这一块不是修补，是一条新的平台能力线。

这三个阶段不是同一种工作。

- 阶段一主要是“把系统跑起来”
- 阶段二主要是“把产品做出来”
- 阶段三主要是“把平台能力补齐”

如果前两阶段 scope 控制得好，前两阶段是可以比较快推进的。真正难的是第三阶段。

## 2. 当前项目对 macOS 的真实状态

当前仓库不是完全 Windows 死绑，但客户端自动化核心明显是 Windows-first。

### 2.1 已经具备一定跨平台基础的部分

- 前端 Electron 打包配置本身已经有 `mac` 目标
- 很多 Python 组件本质上是纯逻辑型或服务型，不依赖 Windows API
- 浏览器插件相关代码已经存在部分 Unix 路径，说明设计上不是完全排斥非 Windows
- `uv`、Python 包管理、后端服务这些基础设施本身都可以跨平台

### 2.2 明显是 Windows-first 的部分

- 当前构建入口是 `build.bat`
- 产品资源打包依赖 Windows 目录结构、`python.exe`、`7z.exe`、`certutil`
- picker / locator / window / word / 部分 system 能力有明显 Windows 实现偏向
- 现有桌面自动化术语和结构主要围绕 UIA / MSAA / Win32 生态展开

### 2.3 一个重要判断

阶段一和阶段二不应该被桌面自动化绑死。

只要我们愿意明确砍掉 Win-only 能力，把浏览器自动化、脚本执行、数据处理、网络请求、文件处理这些能力先跑通，macOS 最小可运行版是有机会在较短时间内做出来的。

## 3. 总体分阶段目标

### 阶段一：最小可运行版本

目标：

- 在 macOS 上构建并启动客户端
- 浏览器自动化链路可用
- 80% 可跨平台组件可以运行
- Win-only 组件明确不可用，而不是随机报错

时间预估：

- 2 周

### 阶段二：可发布版本

目标：

- 覆盖全部可跨平台组件
- 解决 macOS 权限、签名、安装包、发布链路
- 达到可对外发版的质量线

时间预估：

- 1 到 2 周

### 阶段三：桌面自动化

目标：

- 提供 macOS 原生桌面自动化能力
- 支持拾取、窗口定位、元素高亮、基础桌面控件交互

时间预估：

- 暂按 1 个月
- 实际是否够用，要看 Accessibility API 接入后的复杂度

## 4. 阶段一：浏览器自动化最小可运行版本

### 4.1 目标定义

这一阶段不是做“macOS 版完整 RPA”，而是做“macOS 上可工作的浏览器自动化客户端”。

最重要的是尽快形成一条完整闭环：

1. 构建产物
2. 客户端启动
3. Python runtime 可运行
4. scheduler / executor 可运行
5. 浏览器插件可安装 / 检测 / 使用
6. 流程能执行浏览器相关任务

### 4.2 这一阶段建议纳入范围的能力

建议优先保证这些模块：

- `astronverse-browser`
- `astronverse-browser-plugin`
- `astronverse-script`
- `astronverse-dataprocess`
- `astronverse-datatable`
- `astronverse-network`
- `astronverse-openapi`
- `astronverse-pdf`
- `astronverse-report`
- `astronverse-system` 中真正可跨平台的子能力
- `executor` / `scheduler` / `trigger` 中不依赖 Windows 拾取的部分

### 4.3 这一阶段建议明确排除的能力

这些能力不要在阶段一里硬上：

- 桌面 picker
- UIA / MSAA 相关 locator
- Window 组件的完整桌面控制
- Word / Office / COM 自动化
- 明显绑定 Windows 桌面生态的软件自动化
- 企业软件类拾取和控制

阶段一如果把这些一起带上，速度会明显下降，而且会把问题混在一起。

### 4.4 阶段一的关键技术路线

#### 路线 A：先拆出跨平台构建入口

当前 `build.bat` 只是 Windows 构建器，不应该继续作为 macOS 主入口。

阶段一建议直接做一个新的跨平台构建入口，比如：

- `build.py`
- `scripts/build_engine.py`
- `make build-macos`

目标不是一步到位替换全部脚本，而是先把下面几件事做出来：

- 生成 Python runtime 资源
- 打包 engine 依赖
- 生成前端 Electron 所需资源目录
- 在 macOS 上能产出可运行构建结果

重点不是“把 bat 翻译成 sh”，而是把构建逻辑抽成平台无关步骤。

#### 路线 B：先保证 engine 在 macOS 本地能启动

建议不要一上来就跑完整客户端，而是先跑 engine 关键服务：

- executor
- scheduler
- browser bridge
- trigger

如果这些服务在 macOS 下起不来，前端再漂亮也没有意义。

阶段一需要先确认的是：

- 哪些服务 import 即炸
- 哪些服务启动后会因为平台依赖缺失而退出
- 哪些服务只是功能缺失，但仍能正常启动

#### 路线 C：浏览器自动化优先走浏览器插件 + Web locator

浏览器自动化这条线，尽量不要走桌面拾取降级方案。

阶段一要优先保住的能力是：

- 浏览器插件注入
- 浏览器启动 / 检测
- 页面元素定位
- 页面点击 / 输入 / 获取文本
- 页面截图或基础视觉辅助

也就是说，阶段一浏览器自动化的核心路线应该是：

`Electron + Python engine + browser plugin + browser bridge + web locator`

而不是：

`Electron + 桌面 picker + OCR + 鼠标模拟`

后者会把阶段一直接拖进阶段三。

#### 路线 D：Win-only 组件做显式下线

阶段一不能接受“用户点了组件才知道不支持”。

要尽早建立一份 macOS 组件支持矩阵，至少分三类：

- 支持
- 部分支持
- 不支持

前端、引擎、组件元数据里最好都能体现这个信息。哪怕阶段一只是先做运行时拦截，也比静默失败强。

### 4.5 阶段一的核心阻塞点

#### 阻塞点 1：构建入口 Windows-only

当前构建依赖 `build.bat`。这不是小问题，是阶段一的第一 blocker。

如果构建入口不先拆，后续所有验证都会不稳定，且无法进入 CI。

#### 阻塞点 2：system 组件“看似跨平台，实际只有 Win/Linux”

最典型的是：

- clipboard：非 Windows 直接走 Linux 实现
- screenshot：非 Windows 依赖 `scrot`

这说明不能只看依赖包是否支持 macOS，而要看实现分支是否真的有 macOS 路径。

#### 阻塞点 3：浏览器插件支持不完整

浏览器插件代码有 Unix 分支，但当前分支逻辑并没有明确支持 Darwin 全链路。

这一块要尽快验证：

- Chrome / Edge 在 macOS 的插件安装策略是否已经可用
- 插件检测是否依赖 Linux/Windows 的特殊路径
- 浏览器进程识别是否可靠

#### 阻塞点 4：picker / scheduler 里存在平台分支

虽然阶段一不做桌面自动化，但 scheduler 里仍可能在初始化 picker 相关逻辑时触发平台问题。

必须把“浏览器自动化最小闭环”需要的服务启动路径和“桌面拾取服务启动路径”拆开看。

### 4.6 阶段一建议的实际推进顺序

建议按下面顺序推进：

1. 先在 macOS 上让 Python engine 依赖同步成功
2. 确认 executor / scheduler / browser bridge 可以启动
3. 做跨平台构建入口雏形
4. 验证 Electron 客户端能跑、能连 engine
5. 验证浏览器插件安装和 bridge 通讯
6. 冒烟测试跨平台组件
7. 对不支持组件做明确下线

### 4.7 阶段一的验收标准

阶段一至少要满足：

- macOS 上可以构建出可运行客户端
- 浏览器插件可以安装或被系统识别
- 浏览器页面定位、点击、输入、读文本可以跑通
- 至少一组真实流程可执行
- 跨平台组件有清晰支持矩阵
- Win-only 组件不会误入可用路径

## 5. 阶段二：可发布版本

### 5.1 阶段二目标

阶段二不是“多支持几个组件”，而是从“开发可跑”升级到“用户可安装、可使用、可升级”。

重点在产品化问题，而不是纯代码问题。

### 5.2 阶段二的关键技术路线

#### 路线 A：补全 macOS 权限模型

macOS 自动化相关至少会碰到：

- Accessibility
- Screen Recording
- Automation / Apple Events
- 文件系统访问

阶段二要做的不是仅仅调用 API，而是：

- 第一次启动时如何检查权限
- 缺权限时如何引导用户
- 被拒绝后怎么恢复
- 签名后权限行为是否变化

#### 路线 B：补齐签名与打包

只要涉及浏览器插件、自动化、系统权限，签名和 entitlements 基本绕不开。

阶段二至少要确认：

- Electron mac 包能正常签名
- 签名后 Python runtime 和额外资源仍可访问
- DMG / 安装包流程完整
- 打包产物在干净机器可安装

#### 路线 C：跨平台组件清单收口

阶段二要形成一份稳定清单：

- 哪些组件在 macOS 正式支持
- 哪些组件明确不支持
- 哪些组件处于实验状态

这份清单最好和构建、文档、前端展示策略统一，不要三套说法。

### 5.3 阶段二核心问题

- 权限申请和签名的组合问题
- 安装包与更新流程
- 开发机器能跑、发布后不能跑的差异
- 用户第一次使用时的失败体验

### 5.4 阶段二验收标准

- 干净 macOS 机器上可安装
- 首次启动权限引导清晰
- 浏览器自动化链路稳定
- 跨平台组件白名单全部可用
- 文档完整，内部同学能复现构建与安装

## 6. 阶段三：桌面自动化

### 6.1 先做一个现实判断

桌面自动化不是给现有代码加几个 `if sys.platform == "darwin"` 就能完成的。

它本质上是要在 macOS 上重建一条新的桌面能力链：

- 元素拾取
- 元素描述
- 元素高亮
- 窗口控制
- 元素定位
- 输入输出控制

### 6.2 阶段三的核心技术路线

#### 路线 A：基于 macOS Accessibility API 重建 picker / locator

Windows 这边的 UIA / MSAA 思路，不能直接照搬。

macOS 上应该围绕 Accessibility API 做抽象，重点不是复制 Windows 的类型名，而是抽象出平台无关接口，例如：

- `Application`
- `Window`
- `Element`
- `Role`
- `Attributes`
- `Actions`
- `Bounds`

也就是说，阶段三最关键的不是先写代码，而是先把平台无关的元素模型整理出来。

#### 路线 B：先做“可拾取 + 可定位 + 可高亮”，再做复杂动作

建议顺序：

1. 拾取当前元素
2. 显示边框高亮
3. 输出元素路径/属性
4. 再做点击、输入、菜单、表格等复杂操作

不要一开始就试图覆盖所有桌面行为。

#### 路线 C：窗口能力单独抽象

窗口控制和元素定位不要混在一起。

窗口能力先做：

- 查找窗口
- 激活窗口
- 置顶
- 调整大小
- 获取窗口 bounds

这样很多桌面流程即便还没有完整控件拾取，也能先跑一部分。

### 6.3 阶段三主要风险

- macOS Accessibility 数据模型和 Windows 差异很大
- 不同应用的可访问性质量差异非常大
- 高亮与拾取会遇到多显示器、缩放、权限、前后台切换问题
- 企业软件兼容度不可预估

### 6.4 阶段三验收标准

建议把验收拆成三级：

- L1：窗口可控 + 基础元素可拾取
- L2：常见桌面控件可定位与交互
- L3：复杂桌面软件与企业软件场景

阶段三如果不分级，容易一直没有“完成”的感觉。

## 7. 关键技术路线总表

这里把最关键的技术判断单独列一下。

### 7.1 构建路线

不要继续在 `build.bat` 上打补丁来支撑 macOS。

应该尽快形成新的跨平台构建入口，把构建步骤抽成：

- engine 依赖同步
- 组件打包
- Python runtime 资源准备
- 前端构建
- Electron 打包

Windows 后续也应该逐步迁到这套入口上。

### 7.2 浏览器自动化路线

阶段一和阶段二，核心路线必须是：

- 浏览器插件
- browser bridge
- web locator
- executor/scheduler

不要让浏览器自动化依赖桌面拾取。

### 7.3 桌面自动化路线

阶段三必须以平台抽象为前提：

- 平台无关元素模型
- macOS Accessibility 实现
- Windows 实现逐步适配到新抽象

如果不做这一步，后面会出现一堆 `if darwin / if win32` 的碎片逻辑，长期不可维护。

### 7.4 组件支持路线

建议尽早为 macOS 建一份组件矩阵：

- 纯跨平台：优先支持
- 需少量适配：阶段一/二解决
- 明显 Win-only：阶段三前不碰
- 永久不支持或暂不支持：明确写清楚

## 8. 组件分层与优先级

### P0：阶段一必须搞定

- 构建入口
- Electron 客户端启动
- engine runtime 启动
- 浏览器插件
- 浏览器 bridge / web locator
- script / dataprocess / network / openapi / report / pdf
- `system` 中必要且可跨平台的部分

### P1：阶段二补齐

- 权限引导
- 签名 / 打包 / 安装
- 所有可跨平台组件清理完成
- 文档 / CI / 发布流程

### P2：阶段三主战场

- picker
- locator
- window
- word / office 类自动化
- 桌面元素高亮
- 企业软件桌面自动化

## 9. 最容易低估的坑

### 坑 1：以为“依赖能安装”就等于“组件可运行”

不是。很多组件的问题在平台分支实现，不在依赖安装。

### 坑 2：把 Linux 分支当成 macOS 分支

当前很多代码是“非 Windows 就走 Linux”，这在 macOS 上通常是不对的。

### 坑 3：阶段一 scope 漂移

只要阶段一开始尝试桌面自动化，就很容易失控。

### 坑 4：签名前后行为不同

本地开发环境跑通，不代表签名和安装后还能跑通。

### 坑 5：浏览器自动化和桌面自动化边界不清

这一点必须尽早厘清，否则阶段一和阶段三会互相污染。

## 10. 建议的下周启动动作

如果下周一/二开工，我建议第一轮动作就是这些：

1. 在 macOS 本机拉起 `uv` 环境和 engine 关键服务
2. 列出阶段一组件白名单
3. 列出当前会在 macOS import/启动失败的模块
4. 起一个新的跨平台构建入口
5. 验证浏览器插件链路
6. 做第一版 macOS 组件支持矩阵

先把“能不能形成最小闭环”验证出来，再决定哪里需要重构，哪里只需要补平台分支。

## 11. 我当前最强的建议

### 建议 1

阶段一不要追求“尽量多支持组件”，而要追求“形成最短闭环”。

### 建议 2

先把浏览器自动化做成产品，再做桌面自动化，不要反过来。

### 建议 3

桌面自动化必须走抽象层，不要在 Windows 结构上继续横向加平台分支。

### 建议 4

尽早形成一份 macOS 组件支持清单，并让前端、构建、文档统一使用它。

这件事越早做，后面越不容易乱。

## 12. 关于 Linux / 信创系统的策略补充

前期已经做过一些 Linux 和信创系统适配，这是好事，但后续目标最好更收敛，不要把 Linux 路线和 macOS 路线混成一条。

我对 Linux / 信创方向的建议是：

- 前中期只做网页自动化
- 叠加纯跨平台组件能力
- 不把桌面自动化放进主线目标

### 12.1 为什么 Linux 更适合先做网页自动化

Linux 的难点不是“不能自动化”，而是桌面环境差异太大。

真正做桌面自动化时，马上会遇到这些变量：

- 发行版不同
- 显示协议不同：`X11` / `Wayland`
- 桌面环境不同：`GNOME` / `KDE` / 定制环境
- 输入注入和窗口管理能力差异很大
- 信创系统还会叠加国产桌面、安全策略和裁剪环境

这意味着 Linux 桌面自动化不是一个单一目标，而是一整套平台兼容矩阵。

相比之下，网页自动化路径更稳定，核心路线仍然可以是：

- 浏览器插件
- browser bridge
- web locator
- executor / scheduler
- 跨平台组件执行

这一条线在 Linux 上更容易形成稳定交付。

### 12.2 Linux 更合理的产品定义

如果要定义 Linux 版本，我建议先把它理解成：

- 浏览器自动化平台
- 流程编排平台
- 数据与接口自动化平台
- 非桌面控件型 RPA 平台

不要把它定义成“Windows 客户端能力的 Linux 平移版”。

这个预期管理非常重要。否则后面一旦有人把 Linux 和 Windows 做横向对比，团队会被拖进错误的承诺。

### 12.3 Linux 桌面自动化是否完全不做

不是完全不做，而是不应该进入前期主线。

如果后续真要做，建议采用非常受控的策略：

- 固定发行版
- 固定桌面环境
- 固定显示协议
- 固定客户场景

也就是说，Linux 桌面自动化更适合作为研究项、定制项或行业专项能力，而不是一开始就做成通用产品能力。

### 12.4 和 macOS 路线的关系

macOS 和 Linux 不应该共用一套推进假设。

- macOS 的中长期目标可以包含桌面自动化
- Linux 的中长期目标优先级仍然应该是网页自动化与跨平台组件

原因很简单：

- macOS 平台相对统一
- Linux 桌面生态高度碎片化

所以两条路线的目标上限可以不同，不需要强行保持一致。

### 12.5 对团队节奏的实际意义

如果 Linux 先聚焦网页自动化和跨平台组件，你们会得到几个直接收益：

- 更快形成可交付版本
- 更容易支持信创场景
- 更少被桌面环境差异拖住
- 可以把桌面自动化研发资源集中给 macOS

这个取舍不是保守，而是更符合投入产出比。
